# Program Research Notes

These notes summarize public signals observed before submitting a support
application. They are intentionally conservative and should not be treated as a
guarantee of selection.

## Official Criteria Summary

OpenAI's Codex for Open Source materials emphasize active maintainers of public
open-source projects, especially projects with meaningful usage, broad adoption,
ecosystem importance, and real maintenance work such as pull request review,
issue triage, release workflows, and security or code quality work.

## Publicly Visible Accepted Example: Databasus

Databasus publicly states that it was accepted into Codex for Open Source and
Claude for Open Source. Its public repository signals are strong:

- thousands of GitHub stars
- hundreds of forks
- hundreds of commits
- production-focused security and reliability documentation
- explicit AI usage policy requiring human verification
- clear product value: database backup, restore verification, and disaster
  recovery

## Large Maintainer Example: Yii2

Yii2 is a mature framework with years of maintenance history and very high
public adoption. Its repository has thousands of stars and forks, a large issue
queue, and a long-running maintainer workload.

## Maintainer Signal Kit Readiness

Current strengths:

- public MIT-licensed repository
- CI, tests, release tags, governance files, and security policy
- explicit maintainer and architecture documentation
- evidence pack generation
- public GitHub metrics command
- Codex workflow documentation for PR review, issue triage, release prep, and
  security review

Current weaknesses:

- new repository
- no external contributors yet
- no stars, forks, package downloads, or dependent projects yet
- no public issue triage history yet

## Honest Submission Strategy

The strongest truthful framing is not that this repository is already critical
infrastructure. It is that the project directly supports the maintainer work the
program is designed to help: evidence collection, security-aware review, issue
triage, release preparation, and conservative support application drafting.

Do not claim adoption that does not exist. Instead, explain the roadmap and the
maintainer workload the project is built to reduce.
